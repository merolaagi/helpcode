# v0.3.2 Design: the remote sous-chef (first free-tier provider)

**Status:** Design draft. No code. Spec for the v0.3.2 build session.
**Builds on:** v0.3.1 (three local sous-chefs: selection, triage, decomposition).
**The big shift:** this is the first sous-chef that is NOT local. It calls a
remote free-tier API, which introduces credentials, quota, and — the decisive
constraint — a data-privacy boundary.

---

## 1. The decisive constraint: free-tier data is training data

Google's Gemini free tier (and most free tiers) **may use your inputs to train
their models.** Their own guidance: do not send proprietary code or confidential
data through the free tier.

helpcode's published promise is the opposite: "nothing leaves your machine
except what you paste into Claude.ai." A remote sous-chef that ships the user's
source code to a free tier would silently break that promise and could leak
proprietary code into a third party's training set.

**This shapes everything below.** The rule for v0.3.2:

> A remote free-tier sous-chef may only receive data the user has explicitly
> accepted leaving the machine, and by default must NOT receive raw source code.

Concretely, remote sous-chefs are restricted to tasks whose **input is not the
user's proprietary code**, or where the user has explicitly opted in per-project.
See §4 (what remote workers are allowed to do).

---

## 2. Provider choice: Gemini free tier, first

**DECIDED: Gemini free tier is the first provider; pick the exact model at build
time** after re-checking current limits (they change often — Dec 2025 saw 50-80%
quota cuts and models get deprecated). Re-verify RPM/RPD and model availability
with a quick search at the start of the build session, then choose between
Flash-Lite (most throughput, ~15 RPM/1000 RPD) and Flash (better reasoning,
~10 RPM/250 RPD).

Rationale for Gemini as the first provider (as of mid-2026, verify at build):
- No credit card, no expiration — the only major indefinite free tier.
- Flash-Lite: highest free throughput (≈15 RPM, ≈1,000 requests/day), 1M-token
  context. Plenty for prep work.
- Simple REST API; no SDK needed (keeps zero-dependency property).
- 429 + RESOURCE_EXHAUSTED on quota exhaustion, with retry-after headers — clean
  signal to fall back.

The `SousChef` abstraction (below) means adding DeepSeek/Groq/others later is
incremental. Gemini is just the first.

---

## 3. The `SousChef` interface (the real refactor)

Today "local sous-chef" is implicit — selection/triage/decompose each call the
Ollama client directly. v0.3.2 introduces an interface so local and remote
workers are interchangeable and the router can pick among them.

```
interface SousChef {
  id: string;                    // "local:qwen2.5-coder:7b", "gemini:flash-lite"
  kind: 'local' | 'remote';
  // Can this worker take this task given the privacy rules + its quota?
  canHandle(task: SousChefTask, ctx: TaskContext): boolean;
  // Do the work. Throws on failure (caller falls back).
  run(prompt: string, opts): Promise<string>;
  // For remote workers: current free-quota state, for the cockpit + routing.
  quota(): QuotaStatus | null;
}
```

The existing local Ollama calls get wrapped as a `LocalSousChef` implementing
this interface. No behaviour change for local; it just gets a uniform shape.
The new `GeminiSousChef` implements the same interface.

Routing principle (unchanged from the design doc): **cheapest capable worker
first.** local → remote-free → (escalate to principal via paste). A remote
worker is only tried when (a) a local worker can't handle the task or isn't
available, AND (b) the privacy rules permit the data leaving the machine, AND
(c) the remote has free quota left.

---

## 4. What remote workers are allowed to do (privacy-gated)

Given §1, map each sous-chef task to whether a remote free-tier worker may do it:

| Task | Remote by default? | With `allowRemoteCode` opt-in? | Why |
|---|---|---|---|
| file_selection | NO | YES | input includes file contents (proprietary code) |
| output_triage | NO | YES | test output can contain code, secrets, paths |
| decomposition | YES | YES | input is the task *description* the user typed; no source code |
| (future) commit-msg | NO | YES | diff is source code |

**DECIDED (per product philosophy):** ship BOTH the safe default AND an explicit
opt-in.

- **Default:** remote does **decomposition only** — its input is a
  natural-language task description the user was already going to paste into
  Claude. No source code leaves the machine. This is the safe, no-surprise path.
- **Opt-in (`allowRemoteCode`, per-project, off by default):** a consenting user
  can flip this in `project.json` to route code-bearing tasks (selection,
  triage) to the remote free tier too — maximising free-tier leverage, which is
  the whole point of helpcode.

**Why both, not just "allow code by default":** the *user* understanding the
free-tier training tradeoff doesn't mean their *client/employer/collaborators*
do. Someone running helpcode on a work repo under NDA shouldn't have code leave
the machine silently. So code-to-free-tier requires informed consent:

- `allowRemoteCode` defaults to `false`.
- The FIRST time a remote worker would send code (i.e. the first time the opt-in
  takes effect), print a clear one-time notice: this sends your source code to
  Gemini's free tier, which may use it to train Google's models; disable with
  `allowRemoteCode: false`.
- The graduation path this supports: safe default → opt into code-sharing for
  more free leverage → outgrow the free tier → move to paid/different tooling
  when the codebase is valuable enough to protect.

v0.3.2 ships the safe default fully; the `allowRemoteCode` opt-in is included
(it's cheap once the privacy gate exists) but defaults off and is clearly
documented.

## 5. API-key management (must never touch git)

Keys are secrets. The rules:

- **Never in `project.json`** — that file is meant to be committed.
- Store in **`.helpcode/keys.json`**, which `init` adds to `.gitignore`
  (alongside the existing `.helpcode/` ignore — verify keys.json specifically).
- Also support an **environment variable** (`GEMINI_API_KEY`) which takes
  precedence over the file — the standard, CI-friendly, nothing-on-disk option.
- A `helpcode keys` flow (or documented manual step) to set the key. Keep it
  simple: likely just document "put it in env var or .helpcode/keys.json".
- helpcode never logs or prints the key, never includes it in the cockpit,
  never sends it anywhere except the provider's own endpoint.

If no key is configured, the remote sous-chef simply doesn't exist — helpcode
behaves exactly as v0.3.1 (local + deterministic only). Opt-in by presence of a
key.

---

## 6. Quota tracking

Track per-provider usage locally so the cockpit can show it and the router can
avoid exhausted providers:

- Count requests in a rolling window (per-minute and per-day) in state
  (`.helpcode/state.json`, a new `quotas` field, or a separate `quotas.json`).
- On a 429 RESOURCE_EXHAUSTED, mark the provider exhausted until the
  retry-after time; router skips it; cockpit shows it as throttled.
- This is best-effort: we don't perfectly mirror Google's accounting (which is
  per-project, not per-key), but enough to route sensibly and warn the user.

---

## 7. The never-break contract (extended)

Same discipline as every sous-chef so far. The remote worker can fail in more
ways (network, 429, auth, timeout); ALL of them fall back gracefully:

```
remote fails / no quota / no key / privacy-blocked
   → local worker (if it can do the task)
       → deterministic (if any)
           → just proceed without the prep (e.g. plain `ask`)
```

The principal copy-paste flow always works, even with every sous-chef down.

## 8. Build sequence for v0.3.2 (next session)

1. `SousChef` interface + wrap existing local calls as `LocalSousChef`
   (refactor, no behaviour change — existing 101 tests must stay green).
2. `GeminiSousChef` — REST client (no SDK), 429/quota handling, injected fetch
   for tests.
3. Key loading: env var + `.helpcode/keys.json`; ensure gitignored.
4. Privacy gate: the task→remote-allowed map (§4); decomposition is the only
   remote-eligible task in v0.3.2.
5. Router: try local decomposition; if local unavailable AND key present AND
   quota left, try remote; else fall back.
6. Quota tracking + cockpit display (a quota bar for the remote worker).
7. Tests throughout (injected fetch; no live API in CI). Dogfood with a real
   free Gemini key.
8. Docs: README note that an OPTIONAL free-tier key enables a remote sous-chef
   for non-code prep, with the privacy caveat stated plainly.

## 9. Open questions to settle before/while building

- **State vs separate file for quota?** Leaning: a `quotas` field in state.json
  (one source of truth), unless it churns the file too much.
- **How to verify `.helpcode/keys.json` is gitignored** even for users who ran
  an old `init`? Possibly: `helpcode` warns if a keys file exists but isn't
  ignored.
- **Do we even need keys.json, or env-var-only for v0.3.2?**
  **DECIDED: support BOTH** — env var (`GEMINI_API_KEY`, takes precedence,
  nothing on disk, CI-friendly) AND `.helpcode/keys.json` (convenient for users
  who want it persisted per-project). keys.json MUST be gitignored by `init`,
  and helpcode should warn if a keys.json exists but isn't ignored.
- **Privacy disclosure UX:** when a remote worker is first used, print a one-time
  clear notice that the task description is being sent to Gemini's free tier,
  which may use it for training. Consent through configuration, but remind.

---

## 10. Identity check

helpcode's promise stays intact because:
- The **principal** is still copy-paste, no API, no cost.
- Remote sous-chefs are **opt-in (by key presence), free-tier, and forbidden
  from sending source code by default.**
- A user who sets no key gets exactly v0.3.1 behaviour.
- The one thing that can leave the machine to a remote sous-chef in v0.3.2 is a
  **task description the user wrote** — the same text they were about to paste
  into Claude anyway. No code, no surprise.
