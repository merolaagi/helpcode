# v0.3 Design: the "principal + sous-chefs" orchestration model

**Status:** Design draft. No code in this doc — it's the spec the v0.3.x builds work from.
**Supersedes:** the original multi-provider "cockpit" mockup, reinterpreted (see §2).
**Builds on:** v0.2's two-layer model (local Ollama doing file selection + output triage).

---

## 1. The principle (unchanged from day one)

One **principal code helper** does the actual thinking. Everything else — local
models and free-tier remote models — are **sous-chefs**: they prep, trim, parse,
decompose, and pre-digest the work so the principal spends as few tokens as
possible.

- **Principal** = Claude.ai via copy-paste (default), OR a user-chosen model for
  people who prefer another. The principal is where the expensive reasoning
  happens. helpcode does NOT spend principal tokens on plumbing.
- **Sous-chefs** = cheap labour. Local models (Ollama) first; free-tier remote
  APIs (Gemini Flash, DeepSeek, etc.) when a task is too heavy for local.
- **Goal** = minimise principal-helper token consumption. Same goal as v0.1.
  v0.3 just widens the kitchen from one sous-chef (local) to several.

This keeps helpcode's identity: the principal relationship stays copy-paste and
costs nothing beyond the user's existing subscription. Only the *sous-chef* layer
optionally uses (free-tier) APIs.

---

## 2. What we are NOT building, and why (the iframe dead-end)

The original cockpit mockup implied pulling other LLMs' **web interfaces** into
frames and harvesting their conversations to feed the principal. **This is not
viable. Recorded here so it is never revisited:**

1. **iframes are blocked.** ChatGPT, Gemini, Claude, DeepSeek all send
   `X-Frame-Options` / CSP `frame-ancestors` headers that forbid embedding. The
   browser refuses to render them. This is deliberate anti-clickjacking
   protection, not a gap to work around.
2. **Browser-automating the free web tiers violates their ToS** and is brittle
   (bot detection, DOM churn, bans). Not a foundation for a tool others trust.
3. **The legitimate path is the free *API* tiers** — Gemini, DeepSeek, Mistral,
   Groq et al. offer rate-limited free API access that is *allowed*. This needs a
   (free) API key per provider, and is an API call, not a web-session harvest.

So: sous-chefs reach remote models via their **free API tiers**, not by scraping
web UIs. The principal stays copy-paste.

## 3. The two layers, widened

```
        ┌─────────────────────────────────────────────┐
        │  PRINCIPAL  (the brain — expensive, scarce)  │
        │  Claude.ai via copy-paste  [default]         │
        │  or a user-chosen model                      │
        └───────────────▲─────────────────────────────┘
                        │ tight, pre-digested briefs
        ┌───────────────┴─────────────────────────────┐
        │  SOUS-CHEFS  (the kitchen — cheap, plentiful)│
        │                                              │
        │  local (Ollama):   selection, triage,        │
        │                    decomposition, parsing    │
        │  free-tier remote: heavier prep when local   │
        │   (Gemini Flash,   isn't enough; spread load │
        │    DeepSeek, ...)  across providers' free     │
        │                    quotas                     │
        └──────────────────────────────────────────────┘
```

Routing principle: **cheapest capable worker first.** Local before remote;
remote-free before principal. A task only reaches the principal if no sous-chef
can do it well. This is the "free-first, escalate-only-when-needed" idea from the
mockup — but the escalation target is the principal *helper*, and the sous-chef
tier is free.

---

## 4. The cockpit, reinterpreted

The mockup's cockpit is still the right idea — a view of the kitchen — but it
shows **sous-chefs prepping for the principal**, not four paid APIs running
autonomously. What it displays:

- Each sous-chef (local models + any configured free-tier remotes): current task,
  free-quota remaining, recent log lines.
- The principal: what's queued for it, what briefs are ready to paste.
- The pipeline: which prep tasks ran on which worker, what got escalated to the
  principal and why.
- Session metrics: tasks handled locally vs. escalated, tokens saved.

Crucially, the cockpit is a **window onto helpcode's existing work**, not a new
autonomous engine. v0.3.0 builds the window over the TWO sous-chef tasks that
already exist (selection, triage). Later versions add workers and tasks behind
the same glass.

---

## 5. Build sequence (each its own version, dogfooded before the next)

**v0.3.0 — the cockpit view over what exists (FIRST BUILD).**
Make the current local sous-chef visible. A `helpcode status`-style view (or a
`helpcode cockpit` command) that shows: the local model, the two tasks it can do
(selection, triage), what ran this session, and what was escalated to the
principal (i.e. put in a paste block). No new workers, no new APIs. This is the
instrument panel for the engine you already have. It forces us to define the
*data model* of "who did what" cleanly before adding workers.

**v0.3.1 — a third local sous-chef task.**
Add one more local prep task. Strong candidate: **task decomposition** — local
model breaks a big task into ordered sub-steps so the principal gets one focused
brief at a time instead of a sprawling one. (Decomposition is genuinely hard; keep
it conservative — suggest a breakdown for the human to approve, don't act
autonomously.) Surfaces in the cockpit alongside selection and triage.

**v0.3.2 — the remote sous-chef abstraction (one provider).**
Introduce a `SousChef` interface and the first free-tier remote provider
(Gemini Flash free tier is a good first pick: generous free quota, simple API).
A prep task that's too heavy for local can route to the free remote. Per-provider
free-quota tracking. Reuses the v0.2 fallback discipline: remote failure →
local → deterministic. Cockpit shows the remote worker + its quota bar.

**v0.3.3+ — more providers, smarter routing.**
Add DeepSeek, others. A routing policy: cheapest-capable-worker-first with
free-quota awareness. This is where the "complexity score decides the worker"
idea lives — but by now it's incremental, because the worker abstraction and the
cockpit already exist.

---

## 6. The hard questions to resolve per stage (not now)

- **v0.3.0:** what's the data model for "a session's work"? (workers, tasks,
  what-ran, what-escalated, tokens-estimated-saved). Where is it stored — extend
  `.helpcode/state.json`?
- **v0.3.0:** is the cockpit a TUI in the terminal, a `--watch` live view, or a
  static end-of-session summary? Start simplest: a rich `status` printout.
- **v0.3.1:** how does decomposition avoid the autonomous-agent failure mode?
  (Answer leaning: it *proposes*, the human *approves* — never silently acts.)
- **v0.3.2:** API-key management for free-tier providers — where stored, how kept
  out of git, how the user opts in. (Likely: `.helpcode/keys` gitignored, or env
  vars; never in project.json which may be committed.)
- **v0.3.2:** what defines "too heavy for local"? Size? A local-model
  self-assessment? Start with a simple size/timeout heuristic.
- **all:** every remote/local worker keeps the never-break contract: any failure
  falls back to a cheaper-or-deterministic path. The principal copy-paste flow
  must work even if every sous-chef is down.

---

## 7. Identity check (the thing to protect)

helpcode's published promise: the principal relationship is copy-paste, no API
key, no surprise cost. v0.3 must keep that true:

- The **principal** is never auto-called. Always copy-paste (or, for users who
  pick a different principal, their explicit choice).
- Sous-chef APIs are **opt-in, free-tier, and clearly labelled.** A user who adds
  no keys gets exactly today's local+deterministic behaviour.
- The README's "no API key, no surprise costs" stays true for the principal. We
  add a clear note that *optional* sous-chef free-tier keys can be configured.

If a future decision would auto-spend on the principal, that's a different product
and should be named/repo'd separately (the Path 3 we considered and set aside).

---

## 8. Open question carried forward

The original mockup also showed cross-model review (one model checking another's
work). That's a real idea but it belongs late — after the worker abstraction and
routing exist. Parked for v0.3.3+ or later. Not in scope for the first builds.
