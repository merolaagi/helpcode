# v0.2 Design: Local LLM file selection via Ollama

**Status:** Design draft. No code yet. This doc is the spec for the v0.2 build.
**Depends on:** v0.1.3 (current shipped state)
**Scope:** ONE task gets a local-LLM path — file selection. Everything else
stays as-is. Other tasks (response rescue, output triage, commit messages)
are deferred to v0.2.x.

---

## 1. The problem this solves

helpcode's current file selector (`core/selector.ts`) is a keyword heuristic:
filename match + content keyword match + recency. It works, but it's literal.

Example of where it fails: task = "fix the login bug where uppercase emails
fail". The heuristic finds `auth.py` (filename contains nothing, but content
has "login") and maybe `test_auth.py`. It will NOT reason that the bug might
also live in `session.py` or `user_model.py` — because those files don't
contain the literal keywords. A human (or a reasoning model) knows to look
there because they understand *how login works*, not just *which files say
"login"*.

v0.2 adds an optional local-LLM selector that reasons about relevance instead
of just matching strings. It runs on the user's machine via Ollama — no
frontier-model tokens spent, no data leaves the machine.

This is the first concrete instance of the two-layer model from
ARCHITECTURE.md: local compute for plumbing, frontier compute (Claude) for
the actual brain work.

---

## 2. What v0.2 does NOT do

To keep scope tight (the lesson of v0.1):

- Does NOT touch response rescue, output triage, or commit messages — those
  are v0.2.x, after file selection proves the integration.
- Does NOT add embeddings or a vector store — chat-model reasoning only.
- Does NOT make Ollama a hard dependency — the heuristic remains the default
  for anyone without Ollama.
- Does NOT add a new top-level command — file selection happens inside the
  existing `ask` flow.
- Does NOT auto-pull models for the user — we detect and report, never install
  software on someone's behalf.

## 3. The design

### 3.1 Selection strategy: chat-model reasoning

When the local-LLM selector is enabled and Ollama is reachable:

1. Build a candidate list — every source file under `sourceDirs`, with its
   path and a short signature (first ~15 lines, or the list of function/class
   names if cheaply extractable). Cap the candidate list at ~60 files to keep
   the prompt bounded; if a project is bigger, pre-filter with the existing
   heuristic to the top ~60 first.
2. Send the local model a prompt: the task description + the candidate list,
   asking it to return the N most relevant file paths and a one-line reason
   each.
3. Parse the model's response. Validate every returned path against the actual
   candidate list (reject hallucinated paths — this is the critical guard).
4. Return the validated, reasoned selection to the `ask` flow.

If anything fails — Ollama not running, model not found, timeout, unparseable
response, all paths hallucinated — fall back to the existing heuristic
selector silently (with a one-line dim notice).

### 3.2 The local-LLM prompt

The model is asked to respond in a strict, parseable format (same philosophy
as the Claude protocol — lean on format-following):

```
TASK: fix the login bug where uppercase emails fail

CANDIDATE FILES:
- app/auth.py: def login, def logout, def check_password
- app/session.py: class Session, def create, def validate
- app/models.py: class User, class Item
- app/orders.py: def place_order, def cancel
... (up to ~60)

Return the 3-6 files most relevant to the task, most relevant first.
Format each line as:
  PATH | one-line reason
Only use paths from the candidate list above. No prose, no other text.
```

Expected reply:

```
app/auth.py | contains the login function where email is matched
app/models.py | User model likely stores the email field
app/session.py | session creation may normalize the email
```

Parsing is line-based: split on ` | `, validate the left side against the
candidate set. Reject any line whose path isn't a real candidate.

### 3.3 Talking to Ollama

Ollama exposes a local HTTP API at `http://localhost:11434`. We use the
`/api/generate` (or `/api/chat`) endpoint. No SDK dependency — a plain
`fetch` POST keeps helpcode's zero-runtime-dependency property intact.

New module: `core/ollama.ts`

```
interface OllamaConfig {
  enabled: boolean;       // from project.json
  model: string;          // from project.json, e.g. "qwen2.5-coder:7b"
  host: string;           // default "http://localhost:11434"
  timeoutMs: number;      // default 20000 — generous; this is off the
                          // critical path, user is about to read a brief
}

async function isOllamaReachable(host): Promise<boolean>
async function listModels(host): Promise<string[]>   // GET /api/tags
async function generate(host, model, prompt, timeoutMs): Promise<string>
```

Key behaviors:
- `isOllamaReachable` — a fast HEAD/GET with a 1s timeout. If it fails, we
  never even attempt generation; straight to heuristic fallback.
- `generate` — POST with `stream: false` for simplicity in v0.2. Streaming is
  a v0.2.x nicety.
- All errors are caught and turned into "fall back to heuristic" — the LLM
  path can NEVER break the `ask` flow.

### 3.4 The no-Ollama path (non-negotiable)

The selector becomes a strategy chooser:

```
selectFiles(task, config):
  if config.ollama?.enabled:
    if ollamaReachable():
      try:
        result = llmSelect(task, config)   // chat-model reasoning
        if result.length > 0: return result
      catch: (log dim notice)
    # any failure falls through to:
  return heuristicSelect(task, config)      // the current v0.1 selector
```

The existing `core/selector.ts` heuristic stays exactly as it is and becomes
the fallback. We do NOT remove or weaken it. A user with no Ollama, or a
broken Ollama, gets precisely today's behavior.

### 3.5 Configuration (in project.json)

`init` adds an `ollama` block, disabled by default:

```jsonc
{
  "root": "...",
  "language": "python",
  // ... existing fields ...
  "ollama": {
    "enabled": false,                 // off by default; opt-in
    "model": "qwen2.5-coder:7b",      // user edits to a model they have
    "host": "http://localhost:11434",
    "timeoutMs": 20000
  }
}
```

`init` detection logic:
- If Ollama is reachable AND has at least one model pulled, write
  `enabled: true` and set `model` to a detected default (see §3.6).
- If Ollama isn't reachable, write the block with `enabled: false` and a
  comment-like hint so the user knows the option exists.

User turns it on/off by editing `enabled`. No new command needed.

### 3.6 Model choice

User-configurable via `project.json` `ollama.model` (the answer from design
Q3). For the auto-detected default at `init` time:

- Prefer, in order, whichever of these the user already has pulled:
  `qwen2.5-coder`, `codestral`, `llama3.1`, `llama3.2`, `mistral` — coder
  models first since this is a coding task.
- If none of the preferred models are present, use the first model
  `ollama list` reports and note it might not be ideal for code reasoning.
- [TO FILL IN once `ollama list` output is known: name the specific model
  Manish already has, and recommend it as the documented default.]

### 3.7 CLI surface

No new commands. Two new global flags on `ask`:
- `--no-llm` — force the heuristic selector for this one invocation, even if
  Ollama is enabled. (Useful for debugging, or when the user knows the
  heuristic is fine.)
- `--explain-selection` — print the reason each file was chosen (the LLM's
  one-liners, or "matched keyword X / recently modified" for the heuristic).
  This makes the selection legible and builds trust.

## 4. What the user experiences

With Ollama enabled, `helpcode ask "fix login bug"`:

```
• selecting files (local model: qwen2.5-coder:7b)...
✓ selected 3 files:
    app/auth.py        — contains the login function
    app/models.py      — User model stores the email field
    app/session.py     — session creation normalizes email

──────────── COPY EVERYTHING BELOW INTO CLAUDE.AI ────────────
...
```

Without Ollama (or with it disabled), identical to today — the heuristic runs,
no mention of the LLM. No degradation, no nagging.

## 5. Testing strategy

The hard part: the LLM is non-deterministic, so we can't assert on its exact
output. Approach:

1. **Unit-test the parser and the path-validation guard** with canned model
   responses (including adversarial ones: hallucinated paths, prose instead of
   format, empty response). These are deterministic and cover the risky code.
2. **Unit-test the strategy chooser** with a mocked `ollama.ts` — assert it
   falls back to heuristic on every failure mode (unreachable, timeout, throw,
   empty result, all-hallucinated).
3. **Manual dogfood** (Manish, who has Ollama) — run real selections on
   cgm-stats and a bigger project, eyeball quality vs. the heuristic.
4. **NO test depends on a live Ollama** in CI. CI must pass on a machine with
   no Ollama. The LLM path is exercised only via mocks in automated tests.

## 6. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Model hallucinates filenames | Validate every path against the candidate set; reject non-matches |
| Model is slow (big model, no GPU) | 20s timeout; fall back to heuristic; it's off the critical path anyway |
| Model returns prose, not the format | Parse leniently; if <1 valid path, fall back to heuristic |
| Ollama not running | `isOllamaReachable` 1s check before any attempt |
| Large repo blows the prompt | Pre-filter to ~60 candidates with the heuristic first |
| User confusion about whether LLM ran | `--explain-selection` + the "(local model: X)" notice |
| LLM path breaks `ask` entirely | Every failure mode falls back; LLM path is wrapped in try/catch |

## 8. Open questions to resolve before building

1. **`/api/generate` vs `/api/chat`?** Chat endpoint is more standard now and
   handles the system/user split cleanly. Lean chat. Confirm during build.
2. **File signature: first 15 lines, or extracted symbol names?** Symbol names
   are denser/cheaper but need per-language parsing. Start with first 15 lines
   (language-agnostic), revisit if prompts get too big.
3. **Candidate cap of 60 — right number?** Depends on the model's context
   window. Tune during dogfooding. A 7B model with 8k context handles ~60
   short signatures comfortably.
4. **Should `--explain-selection` be default-on?** Trust argument for yes;
   noise argument for no. Tentative: off by default, easily enabled.
